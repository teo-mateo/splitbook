SplitBook Harness

An AI-coding harness driving a local open-weight thinking model (Qwen3.6-27B) to build a medium-sized .NET REST API — reliably.

What's going on

A local model — served by our own vLLM endpoint — is writing a real C#/.NET application end-to-end. The app is a byproduct. The real output is the harness: the set of subagent roles, scoped tools, curated memory, and per-slice protocol that makes a small thinking model deliver correct code without spiraling.

Runtime is opencode — an open-source, Claude-Code-style CLI that lets us define custom subagents in markdown with their own prompts and tool permissions.

Who writes what

primary

builder

Implements the slice. Writes production code. Orchestrates the loop. Invokes subagents by name (@spec-auditor, @test-writer, etc.).

✓ everything

spec-auditor

read-only

Turns the slice's spec into a flat checklist of acceptance criteria. Flags ambiguities. Never designs.

✓ read, grep   ✗ write, bash

test-writer

tests only

Given the checklist, writes failing xUnit tests. Runs dotnet test and confirms red before returning. Never writes production code.

✓ write (tests), dotnet   ✗ edit src/

researcher

on-demand

When any agent is uncertain about a library API or framework behavior, it delegates here. Returns a distilled answer + one code example + source URLs in ~300 tokens. Keeps caller's context lean.

✓ read, webfetch, websearch   ✗ write, bash

reviewer

audit

Audits the slice's diff. Runs tests. Verifies each acceptance criterion has a passing test. Emits a structured {status, findings[]} report. Up to 3 rounds per slice.

✓ read, git, dotnet   ✗ write, edit

lessons-scribe

curator

End of slice: distills ≤3 new lessons into LESSONS.md. Enforces a 20-entry cap by pruning stale ones. Cannot touch any other file.

✓ edit LESSONS.md   ✗ anything else

Per-slice loop

Design choices that matter

These are the non-obvious decisions found the hard way through failure.

Scaffolding before tests, bodies after

The primary may write .csproj, DI wiring, and empty placeholder types before @test-writer runs — but no method bodies, no logic. Real bodies only after tests exist and are red. Prevents tests-written-to-match-existing-code — the #1 small-model TDD failure mode.

Curated lessons, not append-only log

LESSONS.md is capped at ~20 entries. The scribe adds ≤3 per slice and prunes stale ones. Unbounded logs poison the context and get ignored. [HUMAN]-prefixed entries are sacred and take priority.

Research as a subagent, not inline

When the primary is uncertain about a library API, it calls @researcher, which returns ~300 distilled tokens. An inline webfetch would pull 10K tokens of HTML into the primary's conversation and bloat every subsequent turn.

Anti-deliberation protocols

Thinking models spiral. Subagent prompts include hard rules: "if you wrote 'OK let me write now', your NEXT action is a write tool call — not more reasoning". Fixed thinking budgets. Decide-once rules. No re-validating the same classification.

One endpoint per slice

Bigger slices mean bigger batches, more deliberation, and bigger blast radius on failure. Current plan has 17 slices, each 1–2 endpoints. Smaller = more commit-worthy checkpoints, faster recovery.

Output budget matters for thinking models

vLLM max_tokens must be ≥32K. Qwen3's <think> blocks routinely consume 10K–20K tokens before the first tool call. A low ceiling → subagent returns empty, harness protocol breaks silently.

What we're actually measuring

The app is a byproduct. Real outputs:

  • A reproducible harness — prompts, agent definitions, lessons — a fresh operator can rerun.
  • A predictive LESSONS.md — lessons that prevent future failures, not just describe the last one.
  • Data on where this model class breaks — which slice types, how many reviewer rounds, whether lessons reduce future failures, token cost per slice.
